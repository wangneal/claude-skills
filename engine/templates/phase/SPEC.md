| Dimension          | Score | Min  | Status | Notes                              |
|--------------------|-------|------|--------|------------------------------------|

| Boundary Clarity   | 0.80  | 0.70 | ✓ PASS | 范围边界明确，已列出 out of scope |
| Constraint Clarity | 0.75  | 0.65 | ✓ PASS | 技术约束和安全约束已明确          |
| Acceptance Criteria| 0.78  | 0.70 | ✓ PASS | 验收标准可测试，包含 checkbox     |
| **Ambiguity**      | 0.18  | ≤0.20| ✓ PASS | 模糊度低于门控阈值                |

**Scoring Guide:**

- **Goal Clarity (权重 35%):** 目标是否具体、可衡量、可实现、相关、有时限（SMART 原则）
- **Boundary Clarity (权重 25%):** 范围边界是否明确，in/out of scope 是否清晰
- **Constraint Clarity (权重 20%):** 技术约束、安全约束、时间约束是否明确
- **Acceptance Criteria (权重 20%):** 验收标准是否可测试，是否有明确的通过/失败标准

**Ambiguity Calculation:**

```
Ambiguity = 1.0 - Clarity
Clarity = 0.35 × Goal + 0.25 × Boundary + 0.20 × Constraint + 0.20 × Acceptance

Example:
Clarity = 0.35 × 0.85 + 0.25 × 0.80 + 0.20 × 0.75 + 0.20 × 0.78
        = 0.2975 + 0.20 + 0.15 + 0.156
        = 0.8035
Ambiguity = 1.0 - 0.8035 = 0.1965 ≈ 0.20
```

**Gate Check:**
- ✓ Ambiguity ≤ 0.20 (PASS)
- ✓ All dimensions meet minimums (PASS)

## Usage Notes

此模板用于定义可验证的需求规格，确保 WHAT（需求）清晰明确，而非 HOW（实施方式）。

**填写指南:**

1. **Goal 部分**
   - 一个精确的句子，符合 SMART 原则
   - 可衡量、可实现、相关、有时限
   - 提供成功度量标准

2. **Background 部分**
   - 基于代码库的现实情况描述当前状态
   - 列出现有实现、技术栈、已知约束
   - 明确依赖关系

3. **Requirements 部分**
   - 每个需求必须包含 Current/Target/Acceptance 三部分
   - Current: 当前状态、存在的问题、用户痛点
   - Target: 目标状态、新增功能、改进点
   - Acceptance: checkbox 形式的可验证标准

4. **Boundaries 部分**
   - 明确列出 in scope 和 out of scope
   - 对于 out of scope，说明延迟原因和重新考虑的时间

5. **Acceptance Criteria 部分**
   - 所有标准必须可测试（可通过/失败判断）
   - 使用 checkbox 格式
   - 包含功能、性能、安全、兼容性、文档等维度

6. **Ambiguity Report 部分**
   - 对每个维度评分（0.0-1.0）
   - 计算 Ambiguity = 1.0 - Clarity
   - Gate: Ambiguity ≤ 0.20 且所有维度满足最低分
